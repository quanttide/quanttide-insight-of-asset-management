# 陈述型记忆九宫格 — 压力分担模型

## 来源

来自 `quanttide-tech/data/context/insight/asset.md` 的洞察整理。

## 概述

九个数据仓库按压力类型分为三组，每组三个仓库。每组覆盖一个方向的两端，中间由日志驱动串联。

```
                journal（日志）——驱动整个循环
                ┌─────────────────────────────┐
                │        语义类     自我类     │
                │  事件类  │         │         │
                ├─────────┼─────────┼─────────┤
   过去         │  report │ library │ history │  ← 回顾+认同压力
                │  （报告）│ （图书馆）│ （历史）  │
               ─│─────────│─────────│─────────│─
   现在         │ journal │ profile │ brochure│  ← 执行+不确定压力
                │  （日志）│ （档案）  │（宣传册） │
               ─│─────────│─────────│─────────│─
   未来         │ roadmap │ insight │intention│  ← 方向压力
                │  （蓝图）│ （洞察）  │（意图）  │
                ├─────────┴─────────┴─────────┤
                │          AI 执行层           │
                └─────────────────────────────┘
```

## 第一组：未来 — 方向压力

回答「往哪走」。压力源是方向不明确、边界不清楚、判断没依据。

### intention（意图）— 自我类

- **回答的问题：** 为什么做？什么不做？
- **内容特征：**「不做」比「要做」更重要。每一句「我们不做 X」都是帮人类省掉一次纠结。
- **维护方式：** 遇到决策先问「这事在我们的意图范围内吗？」

### insight（洞察）— 语义类

- **回答的问题：** 看到了什么？怎么判断？
- **内容特征：** 记录判断逻辑，而非结论。结论会过时，判断逻辑不会。
- **维护方式：** 做完一件事后，把「我发现了什么规律」写在这里。

### roadmap（蓝图）— 事件类

- **回答的问题：** 去哪里？先去哪再去哪？
- **内容特征：** 可执行的步骤链，每一步应该是「明天可以开始做的事」。
- **维护方式：** 把目标拆成 1-2 天内可完成的步骤，完成即划掉。

## 第二组：现在 — 执行与不确定压力

回答「从哪出发」。压力源是事情太多做不完、外部变化太快、信息不一致。

### journal（日志）— 事件类

- **回答的问题：** 当前在做什么？在想什么？
- **内容特征：** 零门槛的原始记录，整个体系的驱动点。
- **维护方式：** 有想法就记，定期回顾，提炼有持续价值的内容到对应仓库。

### profile（档案）— 语义类

- **回答的问题：** 当前是什么状态？有什么事实依据？
- **内容特征：** 当前状态的事实快照，含可验证数据和明确定义。
- **维护方式：** 以数据（JSON/YAML）为主，关键数据定期更新。

### brochure（宣传册）— 自我类

- **回答的问题：** 对外怎么说？
- **内容特征：** 对外版本的事实和故事。profile 是对内的全部事实，brochure 是对外的精选事实。
- **维护方式：** profile 更新后检查 brochure 是否需要同步。

## 第三组：过去 — 回顾与认同压力

回答「走过什么路」。压力源是不确定自己做对了没有、不知如何证明、怕重复踩坑。

### history（历史）— 自我类

- **回答的问题：** 走过什么路？踩过什么坑？
- **内容特征：** 关键事件时间线 + 经验教训，不记流水账。
- **维护方式：** 定期回顾日志，将有长期价值的事件提炼至此。

### library（图书馆）— 语义类

- **回答的问题：** 外部参考了什么？有什么依据？
- **内容特征：** 外部资料的索引和提炼。价值不在于收藏多少，在于有多少被引用过。
- **维护方式：** 读到的资料记于此，写清楚「为什么值得记」。

### report（报告）— 事件类

- **回答的问题：** 做得怎么样？有什么成果？
- **内容特征：** 结构化的事实汇总，含数据、趋势、对比。
- **维护方式：** 定期从 journal 和 profile 提取数据，生成结构化报告。

## 流转关系

```
journal（日志）——所有想法的入口
    │
    ├──→ 日常执行 → profile 更新当前状态
    │
    ├──→ 发现问题 → intention 调整边界
    │              → insight 记录判断
    │              → roadmap 新增步骤
    │
    ├──→ 完成事项 → report 生成总结
    │              → history 沉淀经验
    │
    ├──→ 外部参考 → library 收录资料
    │
    └──→ 对外输出 → brochure 统一口径
```

压力不是被「解决」的，是被这个系统分流了。每一组承接一部分压力，剩下的交给下一组或 AI。日志是驱动点，AI 是执行层。
