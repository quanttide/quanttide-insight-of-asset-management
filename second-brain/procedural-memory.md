# 程序型记忆九宫格 — 压力分担模型

程序型记忆回答组织「怎么做事」的问题，按约束力层次（宪法、法律、法理）与消费对象（人类友好、AI友好、规则引擎友好）交织为九宫格。

```
                 人类友好          AI友好          规则引擎友好
                  (可读性)        (可解析)          (可执行)
                 ┌──────────────┬──────────────┬──────────────┐
    宪法         │  Bylaw       │ Specification │  Toolkit     │
   （最高约束）    │  工作章程     │  工程标准      │  工具箱       │
                 ├──────────────┼──────────────┼──────────────┤
    法律         │  Handbook    │  Gallery      │  Platform    │
   （中等约束）    │  工作手册     │  工作案例      │  平台         │
                 ├──────────────┼──────────────┼──────────────┤
    法理         │  Tutorial    │  Essay        │  Example     │
   （低约束）      │  工作教程     │  工作札记      │  示例程序     │
                 └──────────────┴──────────────┴──────────────┘
```

## 第一组：宪法层 — 治理压力

回答「底线是什么」。压力源是权责不清、标准缺失、重复劳动。

### Bylaw（工作章程）— 人类友好

- **回答的问题：** 谁做什么？什么能做、什么不能做？
- **压力场景：** 决策时不知道谁有权拍板；跨团队协作时边界模糊；遇到争议没有裁决依据。
- **内容特征：** 治理规则文档，定义宗旨、权力范围和运作原则。所有人必须遵守的基本法。
- **维护方式：** 变更须经审议，保证九宫格结构的一致性与完整性。

### Specification（工程标准）— AI友好

- **回答的问题：** 什么是正确的？怎么判断好坏？
- **压力场景：** AI 生成内容格式不统一、质量参差不齐；团队输出风格不一致；缺乏可量化的质量门禁。
- **内容特征：** 规范化工程实践的文档标准，供 AI 系统解析和验证。定义「什么是正确的」。
- **维护方式：** 在实践中发现问题后迭代完善，提供正确和错误的对比示例。

### Toolkit（工具箱）— 规则引擎友好

- **回答的问题：** 怎么自动化？怎么减少手工？
- **压力场景：** 治理规则靠人记、靠人查、靠人执行，遗漏和出错频繁；重复劳动消耗大量时间。
- **内容特征：** 封装治理规则的自动化工具集，供 CI/CD 和规则引擎直接调用。
- **维护方式：** 随规格标准更新同步迭代，保持工具与规范一致。

## 第二组：法律层 — 操作压力

回答「具体怎么做」。压力源是流程不明确、经验难复用、协同低效。

### Handbook（工作手册）— 人类友好

- **回答的问题：** 这个任务具体怎么做？标准步骤是什么？
- **压力场景：** 新人上手慢，老员工操作靠口口相传；同一个流程不同人做出来不一样；出了错不知道哪个环节有问题。
- **内容特征：** 针对具体岗位或业务流程的标准化操作指导（SOP），只说怎么做，不解释为什么。
- **维护方式：** 引用规范来源，提供版本化链接，更新时同步更新版本号。

### Gallery（工作案例）— AI友好

- **回答的问题：** 以前类似情况怎么处理的？有什么可以参考？
- **压力场景：** 遇到新问题不知道从哪入手；知道有先例但找不到细节；不同项目各自摸索，经验无法复用。
- **内容特征：** 结构化的案例库，供 AI 参考和类比推理。
- **维护方式：** 完成一个阶段后整理案例，标注背景、方案、效果和教训。

### Platform（平台）— 规则引擎友好

- **回答的问题：** 流程怎么跑通？怎么让系统自动运转？
- **压力场景：** 审批靠邮件、沟通靠群聊、进度靠追问；信息散落在各个工具里，找不到统一入口。
- **内容特征：** 承载组织运营、协同办公和业务流转的数字空间，将流程自动化。
- **维护方式：** 持续集成职能域能力，逐步覆盖业务线和职能线。

## 第三组：法理层 — 认知压力

回答「为什么这么做」。压力源是知识门槛高、方法论缺失、理论难落地。

### Tutorial（工作教程）— 人类友好

- **回答的问题：** 这个知识怎么理解？怎么学会？
- **压力场景：** 概念太抽象看不懂；文档只讲规范不讲原理；想学但不知道从哪开始。
- **内容特征：** 面向人的学习材料，帮助理解和掌握操作方法，重视让人理解。
- **维护方式：** 剥离当下的故事性，提炼程序性知识，持续验证可执行性。

### Essay（工作札记）— AI友好

- **回答的问题：** 背后是什么方法论？有什么深层规律？
- **压力场景：** 只知道怎么做，不知道为什么这么做；遇到新问题缺乏判断框架；AI 缺乏足够的方法论上下文。
- **内容特征：** 关于方法论的总结与思考，指导 AI 的上下文构建，偏准学术写作。
- **维护方式：** 围绕现有概念进行整合，逐步交给 AI 接管。

### Example（示例程序）— 规则引擎友好

- **回答的问题：** 这个方案怎么实现？有没有可运行的参考？
- **压力场景：** 理论说得通但不知道怎么编码实现；文档示例与实际情况脱节；重复造轮子。
- **内容特征：** 可执行的代码示例，演示具体实现方式，通常是小型独立项目。
- **维护方式：** 遵循主要语言的项目规范，随技术栈升级同步更新。

## 按约束力层次的纵向压力分流

```
宪法层     定义「什么能做」→ 消除方向性焦虑
  ↑
法律层     指导「怎么做」  → 消除操作性焦虑
  ↑
法理层     解释「为什么」  → 消除认知性焦虑
```

越往上约束力越强，压力也越根本。宪法层解决的是「要不要做」的根本问题，法律层解决「怎么做」的操作问题，法理层解决「为什么这么做」的理解问题。三层叠加，覆盖从决策到执行再到理解的完整压力链。
